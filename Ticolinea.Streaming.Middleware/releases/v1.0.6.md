# v1.0.6 — Stream package priority for local providers

## Summary

Adds a priority order for deciding which package's streams a client receives,
for **local** providers. **External** providers continue to receive all streams.

Priority (local providers):

1. **Client package** — the client's own assigned package
2. **Provider package** — the provider's default package (used when the client has none)
3. **Fallback** — all enabled streams (previous default behavior)

External providers always serve all streams (never filtered).

## Changes in the middleware

- **New** `Helpers/PackageResolver.cs` — single, stateless resolver implementing the
  priority (`external → all`, then `client → provider → all`). Returns an empty
  string to mean "serve all enabled live streams".
- **`Helpers/TokenValidation.cs`** — parses two new JWT claims into
  `TokenValidationResult`: `isExternal` (bool) and `providerPackageId` (string).
  Absent claims default to `false` / `""`, so older tokens behave exactly as before.
- **`Helpers/ClientValidation.cs`** (credential / MAC paths) — the provider join now
  also selects `is_external` and `default_paquete_tv_id`. Uses a **defensive
  fallback**: if those columns don't exist on the node's database, it retries the
  legacy query and degrades to "all streams".
- **`Controllers/StreamsController.cs`** — the 4 modern playlist endpoints
  (`PlaylistByToken`, `PlaylistMobileByToken`, `PlaylistByMac`,
  `PlaylistByCredentials`) now call `PackageResolver.ResolvePaqueteTvId(...)` instead
  of the old `PROVIDER_ID == "main" ? clientPackage : all` line. The
  `PROVIDER_ID == "main"` gate is kept as a safety net for non-main nodes whose DBs
  don't have the Panel-managed package tables.

The legacy `Playlist/{u}/{p}` and `PlaylistMobile/{u}/{p}/{mac}` endpoints
(`Helpers.Usuario` based) are intentionally unchanged.

## Cross-component dependency (Panel API)

This release relies on the Panel API (`ticolinea.panel`) being deployed with:

- New `Provider` fields `IsExternal` and `DefaultPaqueteTvId`, exposed in the
  provider CRUD + admin UI.
- The JWT enriched with `isExternal` and `providerPackageId` claims.
- The EF migration **`AddProviderPriorityFields`** applied to the database
  (`providers.is_external` tinyint(1), `providers.default_paquete_tv_id` varchar).

If the Panel migration is not applied, the new claims are empty and the middleware
safely falls back to "all streams".

## Verification

On a `main` node sharing the database with the Panel, create a local provider with a
default package `PKG_B` and a client:

1. Client assigned `PKG_A` → playlist = `PKG_A` streams (client wins).
2. Client has no package → playlist = `PKG_B` streams (provider default).
3. Client has no package and provider default = NULL → all streams (fallback).
4. Provider `is_external = 1` → all streams regardless of any package.
5. Old token without the new claims → behaves as before.
